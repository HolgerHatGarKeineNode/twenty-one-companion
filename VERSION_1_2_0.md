# VERSION 1.2.0 — "Create & Curate"

> **Mission:** The read-only app (v1.0) becomes a full read-**and-write** app
> with AAA design, a guided onboarding, well-thought-out navigation and CRUD for your own
> meetups, events, places & courses.
>
> **Single Source of Truth.** This file gets ticked off. Read it at session start, find the first
> open `[ ]` checkbox, continue there, set completed items to `[x]`.
> For larger decisions, add an entry to the **Decision Log** (below).

## Starting Point (as of v1.0.x)

- **Stack:** Laravel 13 · Livewire 4 (SFC `⚡*.blade.php`) · Flux UI v2 (free+pro) · Tailwind v4 · NativePHP Mobile v3 · Pest 4.
- **Portal integration:** `app/Services/PortalApi.php` is a **read-only** Saloon facade with a 2-tier cache (fresh + stale/offline). Auth token via the deep-link flow (`PortalAuth`).
- **Pages:** Meetups (list+detail), events, map, courses/lecturers, profile, onboarding (1-step language+region).
- **Navigation:** Bottom nav (Meetups · Events · Map · Profile) + flyout menu (Courses, Lecturers, Cities, Settings).
- **The write API already exists server-side** (Portal): `create/update-meetup`, `add-meetup-to-mine`, `create/update-meetup-event`, `create/update-venue`, `create/update-city`, `create/update-lecturer`, `create/update-course`, `create/update-course-event`. ➜ v1.2.0 makes them usable in the app.

**Guiding principles for v1.2.0**
1. **AAA design** — no generic CRUD look. Motion, haptics, skeletons, empty states with character, consistent brand tokens.
2. **Stay offline-capable** — write operations must coexist with the existing cache/offline model (outbox/retry instead of data loss).
3. **Auth-gated** — writing only with a valid Portal token; clean "please connect" flows.
4. **Every change tested** — Pest feature/Livewire tests, `pint --dirty` before finishing.
5. **Phase-completion ritual** — each phase ends with (a) extending + running the opt-in **integration suite** against the local Portal dev (`scripts/run-integration.sh` with token = including write paths; `composer test:integration` is read-only) and (b) a **Playwright validation** of the new flows **against the local Portal dev** (never the live API — write tests create real records). Both are entered as the final checkboxes of every open phase.
